Standalone Astro blog page process

Use this folder for custom blog articles that need a full Astro layout instead of the Markdown content collection.

Automatic listing

The /blog index automatically scans src/pages/blog/*.astro, excluding index.astro and [...slug].astro.

Every standalone Astro article must define these top-level constants:

---
const title = "Article Title";
const description = "One sentence summary.";
const pubDate = "May 27, 2026";
---

If title or pubDate is missing, the build fails. New standalone Astro pages appear in the Feature Essays section on /blog and in Recent writing on the homepage after npm run build.

Search Console indexing

Astro generates routes for all files in src/pages/blog/*.astro. The @astrojs/sitemap integration includes those routes in dist/sitemap-0.xml because astro.config.mjs sets site: 'https://jackmaguire.org'.

Search Console already has this sitemap submitted:

https://jackmaguire.org/sitemap-index.xml

After adding or changing a standalone Astro article, run:

npm run build
npm run check:indexing

The check confirms that every standalone Astro article in this folder appears on the homepage, appears on /blog, and appears in the generated sitemap. After pushing to main, Vercel deploys the updated sitemap. Search Console discovers the new URL through the submitted sitemap. For urgent posts, inspect the deployed URL manually in Search Console after Vercel finishes.

Essay-style standalone pages (pure prose + citations, no widgets)
Use the .content.md + marked workflow so that long bodies stay in Markdown (with real [text](https://url) links that become <a> on render). 

- Create slug.astro for the shell only (const title/description/pubDate, full <html> chrome with custom <style> for header/prose width if needed, <header class="article-header">, the <div class="wrap"> etc., import and use ReadingProgressBar + Header + Footer, and the <section class="prose" set:html={bodyHtml} />). TableOfContents is OPTIONAL: include it only for long reference or technical articles where section navigation adds real value. For prose essays, omit it and center .wrap directly inside <main> without a flex sidebar wrapper.
- Next to it, create slug.content.md containing the article body written in Markdown (## headings, - lists, [descriptive](https://full-url) for every citation and internal link, **bold** etc.).
- In the .astro (after the consts):
  import { readFileSync } from 'node:fs';
  import { marked } from 'marked';
  const bodyMd = readFileSync(new URL('./slug.content.md', import.meta.url), 'utf8');
  const bodyHtml = marked.parse(bodyMd);
- Then use <section class="prose" set:html={bodyHtml} /> in the markup.
- Widget/ranking pages (those with {data.map}, imported <StocksTable />, <CocktailDatabase /> etc.) continue to author HTML/JSX directly in the .astro; they do not use .content.md.
- This pattern guarantees hyperlinks (no loose URL text) and makes editing long citation sections easy. The check:indexing script now fails the build if it finds bare https?:// in any standalone .astro prose outside href/src.

After changes, always: build + npm run check:indexing (the latter now also enforces no bare URLs).
